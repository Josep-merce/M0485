1. @OneToMany i @ManyToOne

Quan una entitat pot tenir múltiples instàncies d'una altra usem la relació un a molts. Per exemple: un alumne pot tenir moltes matrícules i cada matrícula pertany a un sol alumne.

// Costat "un": Alumne té moltes Matrícules @Entity public class Alumne implements Serializable { @Id @GeneratedValue private Long id; private String nom; @OneToMany(mappedBy = "alumne") private List<Matricula> matricules = new ArrayList<>(); // Constructor, getters, setters... }
// Costat "molts": Matrícula pertany a un Alumne @Entity public class Matricula implements Serializable { @Id @GeneratedValue private Long id; private String modul; @ManyToOne private Alumne alumne; // Clau forana — costat propietari }
📌 mappedBy — qui controla la relació: El costat amb @ManyToOne és el costat propietari (té físicament la referència). El costat @OneToMany(mappedBy=...) és el costat invers. JPA usa el costat propietari per escriure a la BD. El valor de mappedBy és el nom del camp @ManyToOne a l'altra classe.
Sincronització bidireccional

Quan la relació és bidireccional, el programador és responsable de mantenir sincronitzades les dues bandes. JPA no ho fa automàticament:

// MAL: el camp alumne de mat quedarà null Matricula mat = new Matricula("M03"); alumne.getMatricules().add(mat); // Només actualitzem un costat // BÉ: sincronitzem els dos costats Matricula mat = new Matricula("M03"); mat.setAlumne(alumne); // Costat propietari (obligatori per JPA) alumne.getMatricules().add(mat); // Costat invers (per consistència en memòria) em.persist(mat);
JOIN automàtic en JPQL

Quan les entitats estan relacionades, JPQL pot navegar per les relacions sense fer JOINs explícits:

// Tots els alumnes matriculats a M03 List<Alumne> resultat = em .createQuery( "SELECT m.alumne FROM Matricula m WHERE m.modul = 'M03'", Alumne.class) .getResultList(); // JPA genera internament el JOIN necessari — no cal fer-lo a mà

2. @ManyToMany

Quan una entitat pot estar relacionada amb moltes instàncies d'una altra, i viceversa. Per exemple: un alumne pot estar matriculat a molts mòduls i un mòdul pot tenir molts alumnes.

// Costat propietari (sense mappedBy) @Entity public class Alumne implements Serializable { @Id @GeneratedValue private Long id; private String nom; @ManyToMany private List<Modul> moduls = new ArrayList<>(); }
// Costat invers (amb mappedBy) @Entity public class Modul implements Serializable { @Id @GeneratedValue private Long id; private String nom; @ManyToMany(mappedBy = "moduls") private List<Alumne> alumnes = new ArrayList<>(); }
⚠️ Costat propietari a @ManyToMany: El costat sense mappedBy és el propietari. Només les modificacions fetes al costat propietari (afegir a alumne.moduls) es persistiran. Afegir al costat invers (modul.alumnes) no tindrà efecte a la BD.

3. FetchType: LAZY i EAGER

Controla quan es carreguen els objectes relacionats de la BD:

FetchTypeComportamentPer defecte a
LAZYEls objectes relacionats es carreguen quan s'accedeix a la col·lecció per primera vegada.@OneToMany, @ManyToMany
EAGEREls objectes relacionats es carreguen immediatament amb l'objecte pare.@ManyToOne, @OneToOne
// Forçar EAGER en una relació OneToMany @OneToMany(mappedBy = "alumne", fetch = FetchType.EAGER) private List<Matricula> matricules;

4. CascadeType: persistència en cascada

Permet propagar operacions (persist, remove, merge...) des d'una entitat cap als seus objectes relacionats:

// Amb CascadeType.PERSIST: si persistim l'alumne, // les seves matrícules es persisteixen automàticament @OneToMany(mappedBy = "alumne", cascade = CascadeType.PERSIST) private List<Matricula> matricules; // Amb CascadeType.ALL: totes les operacions es propagen @OneToMany(mappedBy = "alumne", cascade = CascadeType.ALL) private List<Matricula> matricules;

Amb CascadeType.PERSIST, el codi del main es simplifica: no cal cridar persist() per a cada matrícula per separat:

// Sense cascada: cal persistir cada objecte manualment em.persist(alumne); em.persist(matricula1); em.persist(matricula2); // Amb CascadeType.PERSIST: només cal persistir l'alumne alumne.getMatricules().add(matricula1); alumne.getMatricules().add(matricula2); em.persist(alumne); // Les matrícules es persiteixen en cascada
💡 Quan usar CascadeType.ALL: Quan el cicle de vida dels objectes fills (matrícules) depèn completament del pare (alumne): si s'elimina l'alumne, també s'haurien d'eliminar les seves matrícules. No useu ALL si els fills poden existir de manera independent.

5. Exercicis

E1 Alumne i matrícules — @OneToMany Bàsic

Implementa un sistema senzill amb les entitats Alumne i Matricula amb una relació @OneToMany / @ManyToOne.

  1. Crea les entitats Alumne (nom, edat) i Matricula (modul, any) amb la relació bidireccional.
  2. Persisteix 3 alumnes. Cadascun amb 2 matrícules. Recorda sincronitzar els dos costats.
  3. Recupera tots els alumnes amb JPQL i mostra el nom i el nombre de matrícules de cadascun.
  4. Recupera les matrícules d'un alumne concret amb find() i navega per getMatricules().
Bàsic CA 8.8
Solució — Alumne i matrícules
@Entity public class Alumne implements Serializable { @Id @GeneratedValue private Long id; private String nom; private int edat; @OneToMany(mappedBy = "alumne", fetch = FetchType.EAGER) private List<Matricula> matricules = new ArrayList<>(); public Alumne() {} public Alumne(String nom, int edat) { this.nom=nom; this.edat=edat; } // getters... } @Entity public class Matricula implements Serializable { @Id @GeneratedValue private Long id; private String modul; private int any; @ManyToOne private Alumne alumne; public Matricula() {} public Matricula(String modul, int any, Alumne alumne) { this.modul=modul; this.any=any; this.alumne=alumne; } }
em.getTransaction().begin(); Alumne a1 = new Alumne("Anna", 19); Matricula m1 = new Matricula("M03", 2024, a1); Matricula m2 = new Matricula("M04", 2024, a1); a1.getMatricules().add(m1); a1.getMatricules().add(m2); em.persist(a1); em.persist(m1); em.persist(m2); em.getTransaction().commit(); // Consultar em.createQuery("SELECT a FROM Alumne a", Alumne.class) .getResultList() .forEach(a -> System.out.println(a.getNom() + ": " + a.getMatricules().size() + " matrícules"));
E2 Autor i llibres — @ManyToMany Mitjà

Implementa un sistema de biblioteca on un autor pot escriure molts llibres i un llibre pot tenir molts autors.

  1. Crea les entitats Autor (nom, nacionalitat) i Llibre (titol, any) amb relació @ManyToMany. Autor és el costat propietari.
  2. Crea almenys 3 autors i 5 llibres. Assigna alguns llibres a més d'un autor i alguns autors a més d'un llibre.
  3. Consulta amb JPQL: llista tots els llibres d'un autor concret (per nom).
  4. Consulta amb JPQL: llista tots els autors d'un llibre concret (per títol).
Mitjà CA 8.8
Solució — Autor i llibres @ManyToMany
@Entity public class Autor implements Serializable { @Id @GeneratedValue private Long id; private String nom, nacionalitat; @ManyToMany // Costat propietari private List<Llibre> llibres = new ArrayList<>(); // constructors, getters... } @Entity public class Llibre implements Serializable { @Id @GeneratedValue private Long id; private String titol; private int any; @ManyToMany(mappedBy = "llibres") // Costat invers private List<Autor> autors = new ArrayList<>(); }
em.getTransaction().begin(); Llibre l1 = new Llibre("Clean Code", 2008); Llibre l2 = new Llibre("Refactoring", 1999); Autor a1 = new Autor("Robert Martin", "USA"); // Afegir al costat PROPIETARI (Autor.llibres) a1.getLlibres().add(l1); a1.getLlibres().add(l2); em.persist(l1); em.persist(l2); em.persist(a1); em.getTransaction().commit(); // Consultar llibres d'un autor List<Llibre> llibresMartin = em .createQuery( "SELECT l FROM Autor a JOIN a.llibres l WHERE a.nom = :nom", Llibre.class) .setParameter("nom", "Robert Martin") .getResultList();
E3 Consultes JPQL amb navegació de relacions Avançat

Partint del projecte de l'E1 (Alumne + Matrícula), implementa consultes JPQL avançades que naveguin per les relacions entre entitats.

  1. Llista els alumnes que no estan matriculats a cap mòdul (sense matrícules). Usa IS EMPTY.
  2. Per a cada mòdul, mostra el mòdul i el nombre d'alumnes matriculats. Usa GROUP BY i COUNT. Ordena per nombre d'alumnes descendent.
  3. Llista els alumnes matriculats al mòdul "M03" usant JOIN explícit a JPQL.
  4. Opcional: Implementa CascadeType.ALL a la relació i comprova que en eliminar un alumne s'eliminen les seves matrícules automàticament.
Avançat CA 8.8
Solució — Consultes JPQL amb relacions
// 1. Alumnes sense matrícules List<Alumne> senseMatricula = em .createQuery( "SELECT a FROM Alumne a WHERE a.matricules IS EMPTY", Alumne.class) .getResultList(); // 2. Alumnes per mòdul (GROUP BY) List<Object[]> perModul = em .createQuery( "SELECT m.modul, COUNT(m) FROM Matricula m " + "GROUP BY m.modul ORDER BY COUNT(m) DESC", Object[].class) .getResultList(); for (Object[] row : perModul) { System.out.println(row[0] + ": " + row[1] + " alumnes"); } // 3. Alumnes de M03 amb JOIN List<Alumne> alumnesM03 = em .createQuery( "SELECT a FROM Alumne a JOIN a.matricules m WHERE m.modul = 'M03'", Alumne.class) .getResultList(); // 4. CascadeType.ALL — eliminació en cascada // A Alumne: @OneToMany(mappedBy="alumne", cascade=CascadeType.ALL) em.getTransaction().begin(); Alumne aEliminar = em.find(Alumne.class, 1L); em.remove(aEliminar); // Elimina l'alumne i totes les seves matrícules em.getTransaction().commit();
