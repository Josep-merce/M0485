A3 — Sistema bancari pas a pas

Construirem junts, pas a pas, un sistema de comptes bancaris que aplica tots els conceptes de l'AEA3. Cada pas afegeix una capa nova de complexitat i explica el perquè de cada decisió de disseny.

Què construirem

Un sistema que gestiona tres tipus de comptes bancaris: corrents, d'estalvis i de crèdit. Al final tindrem una jerarquia completa que usa classes abstractes, herència, polimorfisme i modificadors final.

Com funciona aquesta pràctica: Segueix els passos en ordre. Cada pas inclou el codi complet que has d'escriure, una explicació del perquè de cada decisió i un bloc de prova per verificar que funciona. No avancis al pas següent fins que el codi del pas actual compili i doni la sortida esperada.
Pas 1 Crear la classe base CompteBancari

Comencem creant la classe base. De moment és una classe normal (encara no abstracta). Té el número de compte, el saldo i una quantitat mínima per obrir el compte.

💡 Per què static a qttInicial? Perquè pertany a la classe, no a cada objecte. Tots els comptes comparteixen la mateixa quantitat mínima. Si canvies el valor, canvia per a tots els comptes alhora.
public class CompteBancari { protected String numeroCom; protected double saldo; private static double qttInicial = 100.0; // atribut de classe // Constructor 1: saldo = quantitat mínima public CompteBancari(String num) { this.numeroCom = num; this.saldo = qttInicial; } // Constructor 2: saldo escollit per l'usuari (mínim: qttInicial) public CompteBancari(String num, double saldoInicial) { this.numeroCom = num; this.saldo = (saldoInicial >= qttInicial) ? saldoInicial : qttInicial; } public void imposicio(double q) { saldo += q; } public boolean reintegre(double q) { if (q <= getFondosDisponibles()) { saldo -= q; return true; } return false; } public double getFondosDisponibles() { return saldo; } public double getSaldo() { return saldo; } public String getNumeroCom() { return numeroCom; } @Override public String toString() { return "CompteBancari [" + numeroCom + "] saldo: " + saldo + "€"; } }
✅ Prova-ho: Crea un Main temporal i verifica la sortida esperada:
CompteBancari c1 = new CompteBancari("001"); CompteBancari c2 = new CompteBancari("002", 500.0); c1.imposicio(200); System.out.println(c1); // CompteBancari [001] saldo: 300.0€ System.out.println(c2); // CompteBancari [002] saldo: 500.0€ System.out.println(c1.reintegre(400)); // false: no té prou saldo
Pas 2 Crear CompteCorrent i CompteEstalvis

Ara creem les primeres subclasses. El compte corrent afegeix si s'han emès xecs. El compte d'estalvis afegeix un tipus d'interès compartit per tots els comptes d'estalvis.

💡 Per què final als getters/setters de xecsEmesos i tipusInteres? Perquè no volem que cap subclasse canviï la lògica d'accés a aquests atributs. Els marquem com a final per garantir consistència: ningú els podrà sobreescriure.
public class CompteCorrent extends CompteBancari { private boolean xecsEmesos; public CompteCorrent(String num) { super(num); } public CompteCorrent(String num, double saldo) { super(num, saldo); } // final: cap subclasse podrà sobreescriure aquests accessors public final boolean isXecsEmesos() { return xecsEmesos; } public final void setXecsEmesos(boolean x) { xecsEmesos = x; } @Override public String toString() { return "CompteCorrent [" + numeroCom + "] saldo: " + saldo + "€ | xecs: " + (xecsEmesos ? "Sí" : "No"); } } public class CompteEstalvis extends CompteBancari { // tipusInteres és de classe: tots els comptes d'estalvis el comparteixen private static double tipusInteres = 0.02; public CompteEstalvis(String num) { super(num); } public CompteEstalvis(String num, double saldo) { super(num, saldo); } public static final double getTipusInteres() { return tipusInteres; } public static final void setTipusInteres(double t) { tipusInteres = t; } // calcularInteres és final: la fórmula no es pot canviar public final double calcularInteres() { return saldo * tipusInteres; } public void acumularInteres() { saldo += calcularInteres(); } @Override public String toString() { return "CompteEstalvis [" + numeroCom + "] saldo: " + saldo + "€"; } }
✅ Prova-ho:
CompteCorrent cc = new CompteCorrent("CC-001", 1000); CompteEstalvis ce = new CompteEstalvis("CE-001", 5000); cc.setXecsEmesos(true); System.out.println(cc); // CompteCorrent [CC-001] saldo: 1000.0€ | xecs: Sí ce.acumularInteres(); System.out.println(ce); // CompteEstalvis [CE-001] saldo: 5100.0€
Pas 3 Crear CompteCredit amb getFondosDisponibles()

El compte de crèdit permet fer reintegres que superin el saldo, fins al límit de crèdit. En lloc de sobreescriure directament reintegre(), sobreescrivim getFondosDisponibles() a la subclasse: el mètode reintegre() ja el crida automàticament (el vam dissenyar a Pas 1 pensant en això).

💡 Per què no sobreescrivim directament reintegre()? Si algun dia canviem la lògica de reintegre(), ho fem en un sol lloc. Amb getFondosDisponibles() cada subclasse indica quins fons té disponibles, i reintegre() sempre usa aquest mètode.
public class CompteCredit extends CompteCorrent { private double credit; public CompteCredit(String num, double saldo, double credit) { super(num, saldo); this.credit = credit; } public final double getCredit() { return credit; } public final void setCredit(double c) { this.credit = c; } // Sobreescriu getFondosDisponibles: saldo + crèdit disponible @Override public double getFondosDisponibles() { return saldo + credit; } @Override public String toString() { return "CompteCredit [" + numeroCom + "] saldo: " + saldo + "€ | crèdit: " + credit + "€"; } }
✅ Prova-ho:
CompteCredit cr = new CompteCredit("CR-001", 200, 1000); System.out.println(cr); // saldo 200€ + crèdit 1000€ → podem treure fins 1200€ System.out.println(cr.reintegre(800)); // true System.out.println(cr); // saldo: -600.0€
⚠️ Error típic: Intentar sobreescriure setXecsEmesos() o getCredit() des d'una altra subclasse. Com que estan marcats com a final, el compilador ho impedirà. Això és exactament el que volem.
Pas 4 Convertir CompteBancari en classe abstracta

Ara que tenim les subclasses, convertim CompteBancari en abstracta. Cap usuari hauria de crear un "compte bancari genèric"; sempre crearà un compte corrent, d'estalvis o de crèdit.

💡 Com? Només cal afegir la paraula abstract a la declaració de la classe. No cal canviar res més del codi existent. Aprofitem també per fer que toString() usi getClass().getSimpleName() i pugui ser reutilitzat per les subclasses.
// Únic canvi: afegir 'abstract' a la declaració public abstract class CompteBancari { // ← 'abstract' afegit // ... la resta del codi queda exactament igual ... // toString() millorat: usa getClass() per mostrar el tipus real @Override public String toString() { return getClass().getSimpleName() + " [" + numeroCom + "] saldo: " + saldo + "€"; } }
⚠️ Comprova que ara això dóna error de compilació: CompteBancari c = new CompteBancari("001"); Hauries de veure: "CompteBancari is abstract; cannot be instantiated". Perfecte: significa que la restricció funciona.
Pas 5 Programa principal: polimorfisme en acció

Ara creem el programa principal. Usarem un ArrayList<CompteBancari> (la classe abstracta!) per guardar tots els comptes i veure el polimorfisme en acció.

import java.util.ArrayList; public class Main { public static void main(String[] args) { // Usem la classe abstracta com a tipus de la llista → polimorfisme ArrayList<CompteBancari> comptes = new ArrayList<>(); comptes.add(new CompteCorrent("CC-001", 1500)); comptes.add(new CompteEstalvis("CE-001", 3000)); comptes.add(new CompteCredit("CR-001", 500, 2000)); comptes.add(new CompteCorrent("CC-002", 800)); System.out.println("=== TOTS ELS COMPTES ==="); double totalSaldo = 0; for (CompteBancari c : comptes) { System.out.println(c); // cada toString() és el de la seva subclasse totalSaldo += c.getSaldo(); } System.out.println("Total dipositat: " + totalSaldo + "€\n"); // Accedir a operacions específiques amb instanceof + càsting System.out.println("=== APLICAR INTERESSOS ==="); for (CompteBancari c : comptes) { if (c instanceof CompteEstalvis) { CompteEstalvis ce = (CompteEstalvis) c; System.out.println("Interès generat: " + ce.calcularInteres() + "€"); ce.acumularInteres(); System.out.println(ce); } } System.out.println("\n=== OPERACIONS ==="); comptes.get(0).imposicio(500); boolean ok = comptes.get(2).reintegre(2000); // CompteCredit: usa crèdit System.out.println("Reintegre 2000€ del compte crèdit: " + ok); System.out.println("\n=== ESTAT FINAL ==="); for (CompteBancari c : comptes) System.out.println(c); } }
💡 Observa el polimorfisme: Al bucle for (CompteBancari c : comptes), quan crides c.toString(), Java executa automàticament el toString() de la subclasse correcta — sense cap if/instanceof. Cada objecte sap quin tipus és. Això és el polimorfisme dinàmic en acció.

Jerarquia final de classes

CompteBancari (abstracta) — classe base, no instanciable
   ├── CompteCorrent — afegeix xecsEmesos, accessors final
   │     └── CompteCredit — sobreescriu getFondosDisponibles()
   └── CompteEstalvis — tipusInteres estàtic, calcularInteres() final
